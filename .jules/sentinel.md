## 2025-05-18 - Cryptographically Insecure Randomness in ID Generation

**Vulnerability:** The project uses `Math.random()` to generate unique identifiers in `app/vault.js` (`generateId`), `src/structures/vault/immutable-vault.js` (`generateId`), and `src/information-theory/steganography/chunk-manager.js` (`generateId`). `Math.random()` is predictable and not cryptographically secure, which could allow attackers to predict generated chunk IDs or vault entry IDs, leading to potential data manipulation, collision attacks, or correlation of chunks/vaults.

**Learning:** Developers often reach for `Math.random()` and string conversion methods like `toString(36)` as a quick way to generate identifiers without realizing its unsuitability for security-sensitive contexts, especially since Web Crypto API's `randomUUID` or `getRandomValues` might have historically been harder to use or cross-platform compatibility wrappers were required. The environment memory explicitly states: "Security Learning: When generating IDs or UUIDs in the codebase, always use the globally available Web Crypto API (`crypto.randomUUID()`) or the project compatibility wrapper (`cryptoInstance.randomUUID()`). Do not use `Math.random()`, as it is cryptographically insecure and vulnerable to prediction." and also "Because the project targets Node 22+, the Web Crypto API (`crypto`) is globally available. Consequently, `cryptoInstance` imported from `src/information-theory/cryptography/crypto-compat.js` is populated synchronously and can be used immediately for tasks like `cryptoInstance.randomUUID()` without needing to await an import promise."

**Prevention:** Always use cryptographically secure random number generators (CSPRNG). Specifically, use `crypto.randomUUID()` or the project's `cryptoInstance.randomUUID()` from `crypto-compat.js` whenever generating unique identifiers or UUIDs. Implement a global linting rule or static analysis check to flag usages of `Math.random()`.
